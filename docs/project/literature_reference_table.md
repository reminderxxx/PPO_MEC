# Literature Reference Table

更新日期：2026-05-11

用途：记录与 PPO_MEC 顶刊路线最相关的顶刊/顶会论文，并明确每篇论文能为论文写作提供的参考点，以及 PPO_MEC 相对它的优化点和 claim 边界。

使用规则：

- 本表用于 related work、motivation、baseline rationale 和审稿回应，不是实验结论来源。
- 正式投稿前需要再次核对 IEEE Xplore、ACM DL、USENIX 或会议官网的最终卷期、页码和 BibTeX。
- 不能把外部论文的 service cache、model checkpoint cache、KV cache 直接等同于本项目的 adapter cache；若写作中做类比，必须说明抽象层级差异。
- PPO_MEC 的数值 claim 仍只来自 `artifacts/` 下 paper-grade artifact；本表只说明研究空缺和设计定位。

## 维护规范

每次文献检索或网页浏览时，如果发现与 PPO_MEC 方向相关、且本表尚未记录的论文，必须把它追加进本表。相关方向包括 VEC/MEC task offloading、DAG/dependency-aware workflow、service/model/adapter/KV cache、handoff/migration、multi-timescale control、MARL/DRL baseline、AI serving systems。

新增论文时按以下规范处理：

1. 先查重：用标题、DOI、arXiv ID 或官方页面链接在本文件中搜索；同一论文的 preprint、early access 和正式版本只保留一行，优先使用正式 DOI 或会议/期刊官网链接。
2. 再归类：优先放入现有分区；若现有分区不合适，可以新增二级标题，标题应描述研究线而不是单篇论文。
3. 固定 6 列：`方向`、`论文`、`Venue / Year`、`可提供的参考点`、`PPO_MEC 的优化点 / 差异点`、`论文写作位置`。不要只贴标题或 BibTeX。
4. 要点写法：`可提供的参考点` 写外部论文真实贡献；`PPO_MEC 的优化点 / 差异点` 写本项目相对它的新增问题设定、控制对象、数据链路、机制或 claim 边界；避免把未验证推测写成事实。
5. 信息不完整时保守记录：venue、年份、DOI、页码、acceptance status 或结论没核实时，写 `待核验`；不要为了补齐表格编造。
6. 更新日期：新增或大幅修订本表后，同步更新文件顶部 `更新日期`。
7. 范围控制：无关论文不加入；边缘相关但可能影响审稿回应的论文可以加入，并在 `论文写作位置` 中标注为 `Discussion / reviewer response`。

## 写作 Claim 模板

可采用的谨慎表述：

- Existing VEC studies have investigated service caching, task offloading, migration, and mixed-timescale decisions, but they usually optimize these dimensions separately or for single-shot task/service abstractions.
- Recent LoRA and LLM serving systems show that adapters, checkpoints, and KV caches are first-class scheduling states; however, their setting is mostly data-center serving, not mobility-driven RSU handoff.
- PPO_MEC targets the intersection: continuous DAG workflow execution across RSUs, AI adapter cache cooperation, handoff state migration, and multi-controller RL under a real NGSIM + Alibaba data chain.

不应采用的过度表述：

- 不要声称外部 LoRA/KV-cache 论文已经提供 VEC adapter request trace。
- 不要声称 PPO_MEC 全面优于 oracle prediction 或所有 heuristic upper-bound；当前项目文档已记录 `oracle_prediction` 和 `popularity_cache_heuristic` 边界。
- 不要把 `mappo`、`qmix`、`controller_mat` 写成 vehicle-agent / RSU-agent full MARL；当前都是 controller-level baselines。不要把 `dag_offload_drl`、`cache_offload_drl`、`dt_handoff_drl` 写成主算法变体；它们是领域专项 learned comparators，不能使用 SA-GHMAPPO 专属机制。
